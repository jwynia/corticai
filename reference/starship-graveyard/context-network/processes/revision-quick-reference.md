# Revision Process Quick Reference

## Purpose
At-a-glance reference for the novel revision framework. Keep this open while revising.

## The Five-Step Process

### 1. Identify Revision Type
**Question:** What feels wrong?
- Characters flat/inconsistent → [Character Revision](revision-character.md)
- Pacing/structure issues → [Plot Revision](revision-plot.md)
- Meaning unclear/heavy → [Theme Revision](revision-theme.md)
- Multiple issues → Start with most fundamental

### 2. Analyze Before Changing
**Required:** [Pre-change Analysis](revision-analysis.md)
- Map current state
- Predict consequences (immediate/medium/long-term)
- Set monitoring criteria
- Define rollback triggers

### 3. Implement Systematically
**Process:** [Implementation Workflow](revision-implementation.md)
1. Make minimal viable change
2. Monitor immediately
3. Document observations
4. Compare to predictions

### 4. Manage Cascades
**Track:** [Cascade Management](revision-cascades.md)
- Identify all ripple effects
- Classify by priority (Critical/Important/Enhancement)
- Execute systematically
- Monitor completion

### 5. Document Everything
**Templates:** [Revision Records](revision-records.md)
- Change summary
- Cascade tracking
- Outcome assessment
- Lessons learned

## Warning Signs to Stop

### Critical (Stop Immediately)
- [ ] Character acts impossibly out of character
- [ ] Plot logic completely breaks
- [ ] Timeline becomes impossible
- [ ] Core story purpose lost

### Major (Address Within 24 Hours)
- [ ] Multiple monitoring warnings triggered
- [ ] Cascade tasks multiplying rapidly
- [ ] Original problem getting worse
- [ ] New problems exceeding solutions

## Daily Checklist

Before starting each day:
- [ ] Review yesterday's changes
- [ ] Check monitoring criteria
- [ ] Identify today's focus
- [ ] Set realistic goals

End of each day:
- [ ] Document all changes made
- [ ] Update cascade task list
- [ ] Note any concerns
- [ ] Plan tomorrow's work

## Emergency Procedures

### If Overwhelmed
1. Stop making changes
2. Document current state
3. Take a break
4. Review original goals
5. Consider smaller scope

### If Lost
1. Re-read [pre-change analysis](revision-analysis.md)
2. Check against original problem
3. Review success criteria
4. Trace changes made
5. Consider rollback

### If Rollback Needed
1. Stop all work
2. Document why failed
3. Identify salvageable elements
4. Revert to stable version
5. Plan new approach

## Key Questions

Ask yourself regularly:
1. Am I solving the original problem?
2. Are new problems worth the improvement?
3. Is the story getting better?
4. Am I maintaining what already works?
5. Should I stop and reassess?

- **Documentation:** Constant, not optional
